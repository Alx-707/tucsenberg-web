# Socratic Questioning（苏格拉底提问法）

当用户给出模糊需求时，用此方法论引导用户澄清——而非直接假设答案。

## 触发条件

- 用户需求含歧义词（"优化"、"改进"、"更好"、"帮我做个..."）
- 需求边界不清晰（"加个功能"、"帮我做个页面"）
- 涉及多种可能实现路径
- 用户表示"不懂代码"或"不懂设计"

## 六大原则

| 原则 | 规则 | 示例 |
|------|------|------|
| **单问题** | 每条消息只问一个问题 | ❌ "你想要什么样式？用什么技术？" ✅ "你倾向哪种视觉风格？" |
| **多选优先** | 能用选择题就不用开放题 | ✅ "A. 暗色主题 B. 亮色主题 C. 跟随系统" |
| **探索优先** | 给 2-3 方案 + 权衡分析 | ✅ "方案 A: 简单但灵活性低；方案 B: 复杂但可扩展" |
| **推荐在前** | 推荐方案排第一 + 解释理由 | ✅ "推荐 A（理由：...），备选 B、C" |
| **渐进验证** | 复杂设计分 200-300 字小段逐一确认 | ✅ "先确认架构 → 再确认组件 → 最后确认样式" |
| **YAGNI 无情** | 主动砍掉"可能需要"的功能 | ✅ "这个功能现在真的需要吗？" |

## 提问模板

### 目的澄清

- "你希望这个功能解决什么问题？"
- "如果只能选一个最重要的目标，是什么？"
- "成功的标准是什么？用户会看到什么变化？"

### 约束探查

- "有没有必须兼容的现有系统？"
- "有没有时间/性能/成本约束？"
- "有没有必须遵守的设计规范或品牌指南？"

### 方案选择

- "这里有 2-3 种做法：[A] / [B] / [C]。推荐 A 因为...。你倾向哪个？"
- "方案 A 更简单但扩展性差；方案 B 复杂但灵活。你更看重哪个？"

### YAGNI 检查

- "这个功能是现在就需要，还是'以后可能需要'？"
- "如果去掉这个功能，核心价值会受影响吗？"
- "能否先做最小版本，验证后再扩展？"

## 渐进验证流程

复杂设计必须分段呈现，每段 200-300 字：

```
1. 架构概览（200-300 字）→ 用户确认 ✓
2. 核心组件设计（200-300 字）→ 用户确认 ✓
3. 数据流与状态管理（200-300 字）→ 用户确认 ✓
4. 边界情况与错误处理（200-300 字）→ 用户确认 ✓
```

**回溯规则**：如果用户在任何阶段不同意，立即回溯修订该段，不继续推进。

## 输出物

- 澄清后的需求边界（无遗漏、无冗余）
- 用户确认的方案选择
- YAGNI 裁剪记录（砍掉了什么、为什么）

## 工具集成

使用 `AskUserQuestion` 工具呈现多选问题：
- 最多 4 个选项
- 推荐选项标注 `(Recommended)` 并排第一
- 必要时启用 `multiSelect: true`
